📥 Flujo Completo de un Mensaje Inbound

Cada paso que ocurre desde que el contacto envía un mensaje hasta que recibe respuesta (o no, si está inactivo).

1
WhatsApp → Kapso → Bridge (Node.js)
El usuario envía un mensaje. Kapso recibe el webhook de Meta y lo envía al kapso-bridge/server.mjs. El bridge agrupa mensajes del mismo hilo y enriquece el payload (adjunta transcripciones de audio, URLs de media).
kapso-bridge server.mjs
⚠ Sin escritura a BD todavía
2
Señales de Presencia WhatsApp
El bridge llama al agente FastAPI. Solo si la respuesta NO viene suprimida se envía markRead (visto ✓✓) al usuario. Si el contacto está inactivo, no se envía ninguna señal de presencia.
markKapsoAsRead() suppress_send check
3
FastAPI — Validación y Resolución
Verificación del token interno. Búsqueda del número en wp_numeros (primero por telefono, luego por id_kapso). Si no existe → HTTP 404, mensaje descartado.
kapso_routes.py
Lectura BD
wp_numeros wp_agentes
Escribe debug
debug_events → inbound_received
4
Upsert de Contacto
Busca el contacto por teléfono en wp_contactos. Si no existe, lo crea con origen = "Whatsapp". Si existe, actualiza ultima_interaccion. Devuelve el contacto_id y si fue creado nuevo.
upsert_contacto_canal()
INSERT / UPDATE
wp_contactos
Campos insertados (nuevo contacto)
telefono, empresa_id, origen = "Whatsapp", fecha_registro, ultima_interaccion, is_active = true
Escribe debug
debug_events → contacto_creado / reutilizado
5
Resolución / Creación de Conversación
Busca una conversación activa para ese contacto_id + numero_id. Si no existe, crea una nueva. Verifica que el agente de la conversación pertenezca a la misma empresa.
get_conversacion_activa() insertar_conversacion()
INSERT (si nueva)
wp_conversaciones
contacto_id, agente_id, empresa_id, numero_id, canal = "whatsapp"
6
🔴 Guard: Contacto Inactivo (is_active = false)
Si wp_contactos.is_active = false, el mensaje se guarda en BD con status = "enviado" pero NO se procesa ni responde. El bridge ya no envió visto ni escribiendo (ver paso 2). Solo false explícito bloquea — null no bloquea.
bloqueo mensaje guardado sin respuesta sin visto WA
INSERT mensaje inbound
wp_mensajes → status = "enviado"
Escribe debug
debug_events → inbound_blocked (contacto_inactivo)
7
Procesamiento de Multimedia
Según el tipo de mensaje se procesa de forma diferente. Ver sección Multimedia para el detalle completo.
audio → transcripción imagen → visión AI documento → referencia URL
Supabase Storage (async)
bucket: multimedia wp_multimedia (vía Edge Functions)
8
INSERT Mensaje Inbound en BD
El mensaje del usuario se guarda en la BD con status = "buffer". Si tiene multimedia, el contenido ya incluye transcripción/descripción.
insertar_mensaje()
INSERT
wp_mensajes → status = "buffer"
conversacion_id, contenido, remitente = "usuario", tipo, status = "buffer", metadata.canal, metadata.kapso_message_id, metadata.message_type, empresa_id
9
UPDATE → status = "procesando"
Antes de invocar los agentes, el mensaje se actualiza a procesando para indicar que está siendo procesado activamente.
UPDATE
wp_mensajes → status = "procesando"
10
Ejecución Paralela de Agentes (LangGraph)
Tres agentes corren en paralelo. El conversacional genera la respuesta. El de embudo analiza en qué etapa está el contacto. El de contacto extrae y actualiza datos del contacto.
Conversacional (90s timeout) Funnel Agent (25s timeout) Contact Update (20s timeout)
Lecturas BD (contexto para el agente)
wp_contactos wp_conversaciones wp_mensajes (historial) wp_empresa_perfil wp_empresa_embudo wp_citas wp_contactos_nota
Posibles escrituras (por herramientas del agente)
wp_contactos → etapa_embudo wp_contactos → metadata wp_contactos → es_calificado wp_contactos_nota (guardar_nota)
11
INSERT Mensaje de Respuesta + UPDATE inbound
La respuesta del agente se guarda. El mensaje inbound pasa a enviado. Si hubo error, pasa a error con detalles en metadata.processing_error.
INSERT respuesta
wp_mensajes → remitente="agente", status="sent", modelo_llm, uso_herramientas
UPDATE inbound
wp_mensajes → status = "enviado" / "error"
12
Respuesta al Bridge → WhatsApp
FastAPI devuelve KapsoInboundResponse al bridge. El bridge envía el mensaje al usuario via Kapso API. Si suppress_send=true, el bridge no envía nada.
KapsoInboundResponse dispatchKapsoResponse()

📊 Estados de wp_mensajes

Status Quién lo asigna Cuándo Descripción
buffer insertar_mensaje() Al recibir el mensaje inbound Mensaje recibido, en cola de procesamiento. Todavía no lo ve ningún agente.
procesando _update_inbound_message_statuses() Justo antes de invocar _run_both_agents() Los agentes LangGraph están procesando activamente el mensaje.
enviado actualizar_mensaje() Después de que el agente responde exitosamente El mensaje fue procesado y la respuesta fue enviada al usuario.
enviado (inactivo) insertar_mensaje() directo Contacto con is_active=false — retorno temprano Mensaje guardado normalmente pero sin procesar ni responder. No hay visto ni typing en WhatsApp.
sent insertar_mensaje() Mensaje de respuesta del agente Mensaje de salida (remitente="agente") guardado con este status.
error actualizar_mensaje() con metadata de error Excepción durante procesamiento del agente El agente falló. metadata.processing_error contiene type, detail, failed_at.

Mensajes "stuck" (atascados)

La función get_stuck_messages() detecta mensajes que llevan más tiempo del normal en estado procesando. Esto indica un agente colgado o timeout no manejado.

metadata.processing_error (status = error)

type Tipo de error (timeout, agent_error, etc.)
detail Mensaje de error completo
failed_at Unix timestamp del fallo

🎵 Pipeline de Multimedia

Flujo de audio, imágenes y documentos desde que llegan del usuario hasta que quedan persistidos.

🎤 Audio (.ogg / .mp3)

1Kapso enriquece el mensaje con URL + Transcript en el texto
2_parse_audio_content() extrae audio_url, transcript, filename
3_upload_audio_to_storage() descarga de Kapso URL y sube a bucket multimedia
4message_parts se reemplaza con transcript como texto + storage_url
5[ASYNC fire-and-forget] _multimedia_pipeline():
→ Edge Fn crear-multimedia-inicial-v1
→ Edge Fn guardar-multimedia-v4
Datos guardados en wp_mensajes
contenido = transcript del audio
tipo = "audio"
metadata.storage_url = URL Supabase

🖼️ Imagen (.jpg / .png / .webp)

1_parse_media_content() extrae image_url y filename del texto Kapso
2_upload_audio_to_storage() (reutilizado) sube imagen al bucket
3_describe_image_with_vision() llama a Gemini Flash via OpenRouter (SÍNCRONO, máx 1024 tokens, 3 reintentos con backoff)
4Semaphore limita a 3 llamadas vision concurrentes
5Si vision falla → fallback: "El contacto envió una imagen. URL: ..."
6[ASYNC] _multimedia_pipeline() persiste en wp_multimedia
Datos guardados en wp_mensajes
contenido = "[Descripción de la imagen]: ..." + caption original
tipo = "image"
metadata.storage_url = URL Supabase

📄 Documento (.pdf / .doc / .xlsx)

1_parse_media_content() extrae doc_url y filename
2_upload_audio_to_storage() sube al bucket multimedia
3Construye texto de referencia: "El contacto envió un documento: {filename}\nURL: {storage_url}"
4[ASYNC] _multimedia_pipeline() → wp_multimedia
Datos guardados en wp_mensajes
contenido = referencia del documento
tipo = "document"
metadata.storage_url = URL Supabase
wp_multimedia Vía Edge Functions (async, fire-and-forget)

Edge Function: crear-multimedia-inicial-v1

POST con { contacto_id, tipo, archivo_url }
Crea registro inicial. Devuelve { data: { id: multimedia_id } }

Edge Function: guardar-multimedia-v4

POST con { multimedia_id, contenido }
Guarda transcripción/descripción final en el registro creado.

Supabase Storage Bucket
Bucket: multimedia
Naming audio: audio_{message_id}_{ts}.ogg
Naming imagen: image_{message_id}_{ts}.jpg
Naming doc: doc_{message_id}_{ts}.pdf

🌉 Kapso Bridge (Node.js)

El bridge es el intermediario entre Kapso/WhatsApp y FastAPI. No escribe directamente a Supabase.

📥 Recepción de Mensajes
Webhook de Kapso
  • ✓ Verificación de firma HMAC del webhook
  • ✓ Deduplicación por message_id (TTL 10 min)
  • ✓ Cola por hilo (threadQueues) — mensajes del mismo número van en serie
  • ✓ Enriquecimiento de media por Kapso (transcripciones, URLs)
  • ✓ Timeout de procesamiento: 180s
📤 Envío de Respuestas
Kapso WhatsApp Cloud API
  • ✓ Retry con backoff: máx 3 intentos
  • ✓ Rate limit: 2s base delay
  • ✓ suppress_send=true → NO envía nada (contacto inactivo)
  • ✓ Soporte para reacciones emoji
  • ✓ Delay anti-colisión entre mensajes (1.5s)
👁️ Señales de Presencia
Seen ✓✓ y Escribiendo...
  • ✓ markKapsoAsRead() — solo si respuesta NO suprimida
  • ✓ Contacto inactivo → cero señales de presencia
  • ✓ Antes: seen+typing se enviaban antes del agente
  • ✓ Ahora: seen solo si hay respuesta real
