Endpoints de Envío Manual de Mensajes

Referencia completa para los endpoints /send — GHL y ManyChat
Base URL: https://brain.urpeailab.com  ·  Versión 1.0  ·  Abril 2026

1. Descripción General

El sistema URPE AI Lab expone dos endpoints de envío manual que permiten enviar mensajes directamente a un contacto sin activar el agente de IA. Son útiles para respuestas de soporte humano, notificaciones automáticas externas, o cualquier caso donde se necesite enviar un mensaje programáticamente.

Ambos endpoints comparten la misma filosofía:

1. Recibe contacto_id
→
2. Busca en Supabase
→
3. Recupera credenciales
→
4. Envía mensaje
→
5. Guarda en DB
Dato clave: Solo necesitas enviar el contacto_id entero de Supabase (tabla wp_contactos). El sistema recupera automáticamente el subscriber_id/contact_id de la plataforma, el api_key y el canal desde la última conversación activa en la base de datos.
Prerequisito: El contacto debe haber enviado al menos un mensaje previo al sistema para que exista una conversación con las credenciales almacenadas. Sin historial previo, el endpoint devuelve error 400.

2. Autenticación

Todos los endpoints /send requieren el header X-Send-Key con el valor de la variable de entorno SEND_API_KEY configurada en el servidor.

HeaderValorDescripción
X-Send-Key {SEND_API_KEY} Clave secreta configurada en la variable de entorno del servidor. Si no está configurada, el endpoint es abierto.
Content-Type application/json Requerido en todas las peticiones POST con body JSON. 
# Ejemplo con autenticación
curl -X POST https://brain.urpeailab.com/api/v1/ghl/send \
  -H "Content-Type: application/json" \
  -H "X-Send-Key: tu_clave_secreta" \
  -d '{"contacto_id": 12345, "mensaje": "Hola!"}'

3. Endpoint GHL — Instagram y Facebook

POST https://brain.urpeailab.com/api/v1/ghl/send GHL · IG + FB

Envía un mensaje directo a un contacto de Go High Level (GHL). Compatible con Instagram y Facebook. El canal se detecta automáticamente desde la conversación más reciente en la base de datos.

Body de la petición

CampoTipoReq.Descripción
contacto_id integer Requerido ID entero del contacto en la tabla wp_contactos de Supabase.
mensaje string Requerido Texto del mensaje a enviar. Soporta split de burbujas con \n\n---\n\n entre mensajes.
location_id string Opcional GHL location ID. Si se omite, se recupera automáticamente desde el metadata de la última conversación.

Respuesta exitosa (200)

CampoTipoDescripción
okbooleantrue si el mensaje fue enviado correctamente.
contacto_idintegerEl mismo contacto_id recibido.
contact_idstringGHL contact_id recuperado desde wp_contactos.subscriber_id.
guardado_en_dbbooleantrue si el mensaje quedó registrado en wp_mensajes.
errorstring | nullMensaje de error en caso de fallo. null si todo fue bien.

Ejemplo — Instagram (contacto_id conocido)

curl -X POST https://brain.urpeailab.com/api/v1/ghl/send \
  -H "Content-Type: application/json" \
  -H "X-Send-Key: tu_clave_secreta" \
  -d '{
    "contacto_id": 285318,
    "mensaje": "Hola! Te escribimos para recordarte tu cita de mañana a las 10am."
  }'

Ejemplo — Mensaje con múltiples burbujas

curl -X POST https://brain.urpeailab.com/api/v1/ghl/send \
  -H "Content-Type: application/json" \
  -H "X-Send-Key: tu_clave_secreta" \
  -d '{
    "contacto_id": 285318,
    "mensaje": "Primera burbuja de mensaje.\n\n---\n\nSegunda burbuja de mensaje."
  }'
El separador --- (con saltos de línea alrededor) divide el mensaje en múltiples burbujas independientes enviadas en secuencia. Cada burbuja queda registrada por separado en la base de datos.

Respuesta de ejemplo

{
  "ok": true,
  "contacto_id": 285318,
  "contact_id": "ghl_contact_abc123",
  "guardado_en_db": true,
  "error": null
}

4. Endpoint ManyChat — Instagram y Facebook

POST https://brain.urpeailab.com/api/v1/manychat/send ManyChat · IG + FB

Envía un mensaje directo a un suscriptor de ManyChat. Compatible con Instagram y Facebook. El canal y el subscriber_id de ManyChat se recuperan automáticamente desde la conversación más reciente en la base de datos.

Body de la petición

CampoTipoReq.Descripción
contacto_id integer Requerido ID entero del contacto en la tabla wp_contactos de Supabase.
mensaje string Requerido Texto del mensaje a enviar.

Respuesta exitosa (200)

CampoTipoDescripción
okbooleantrue si el mensaje fue enviado correctamente.
contacto_idintegerEl mismo contacto_id recibido.
subscriber_idstringManyChat subscriber_id recuperado desde wp_contactos.subscriber_id.
guardado_en_dbbooleantrue si el mensaje quedó registrado en wp_mensajes.
errorstring | nullMensaje de error en caso de fallo. null si todo fue bien.

Ejemplo — Facebook (ManyChat)

curl -X POST https://brain.urpeailab.com/api/v1/manychat/send \
  -H "Content-Type: application/json" \
  -H "X-Send-Key: tu_clave_secreta" \

5. Códigos de Error

HTTPCódigo / CondiciónDescripciónSolución
401 X-Send-Key inválida El header X-Send-Key está ausente o es incorrecto. Verificar el valor de SEND_API_KEY en el servidor.
404 Contacto no encontrado El contacto_id no existe en wp_contactos. Verificar que el contacto_id sea correcto.
400 Sin subscriber_id El contacto existe pero no tiene subscriber_id registrado. El contacto debe haber iniciado conversación primero.
400 Sin api_key No se encontró el api_key de GHL o ManyChat en la conversación. El contacto debe haber enviado al menos un mensaje previo.
200 ok: false Error de envío El sistema procesó la petición pero la API de GHL/ManyChat devolvió error. Revisar el campo error en la respuesta para diagnóstico.
Nota: Un HTTP 200 con "ok": false indica que el sistema procesó correctamente la petición pero el envío externo (GHL o ManyChat API) falló. Revisar el campo error para diagnóstico.

6. Flujo Interno Detallado

Este es el proceso interno que ejecuta el sistema al recibir una petición /send:

#PasoDescripción
1Auth checkValida X-Send-Key contra SEND_API_KEY.
2Lookup contactoBusca en wp_contactos por contacto_id. Obtiene subscriber_id y empresa_id.
3Lookup conversaciónBusca la conversación más reciente del canal correspondiente (ghl_* o manychat) en wp_conversaciones.
4Recuperar credencialesExtrae el api_key, location_id (GHL) y canal desde el metadata de los mensajes de esa conversación.
5Split de burbujas(Solo GHL) Divide el mensaje por el separador --- en múltiples burbujas.
6Enviar a API externaLlama a la API de GHL Conversations o ManyChat sendContent con las credenciales recuperadas.
7Guardar en DBRegistra el mensaje en wp_mensajes con remitente="agente", status="enviado" y envio_manual=true en metadata.

8. Variables de Entorno Requeridas

VariableDescripciónObligatoria
SEND_API_KEY Clave de autenticación para todos los endpoints /send. Si no se configura, los endpoints quedan abiertos. Recomendada
GHL_API_KEY Fallback global de GHL API key. Solo se usa si no se encontró el api_key en la conversación de DB. Opcional
SUPABASE_URL URL del proyecto Supabase donde están las tablas wp_contactos, wp_conversaciones, wp_mensajes. Requerida
SUPABASE_SERVICE_KEY Service role key de Supabase para acceso completo a las tablas. Requerida
