URPE AI Lab — Arquitectura del sistema
← Kapso Todos los canales
Stack: FastAPI + LangGraph + Supabase + Express bridge + n8n + OpenRouter (Grok 4.1 fast).
Despliegue dual: Bridge Express en :3001 + FastAPI Uvicorn en :8000, ambos en el mismo contenedor Docker.
Cómo leerlo: click en cualquier diagrama para verlo grande con zoom. Usa el filtro lateral para saltar a secciones.

2. Flujo end-to-end de un mensaje (WhatsApp) sequence

Recorrido completo de un mensaje desde que el usuario escribe en WhatsApp hasta que recibe respuesta. Las fases 1 (funnel + contact_update) corren en paralelo; la fase 2 (conversational) corre después con el resultado de la fase 1.

/kapso/inbound participant DBG as kapso_debug participant SUP as Supabase participant FN as Funnel Agent participant CU as Contact Update participant CA as Conversational participant OR as OpenRouter participant N8 as n8n / Nylas participant SND as WhatsApp sender U->>META: Mensaje META->>KP: webhook KP->>BR: POST + secret BR->>BR: enriquece (vision/audio) BR->>BR: encola (threadQueue) BR->>FA: POST inbound FA->>DBG: stage=inbound_received DBG-)SUP: insert debug_events par Phase 1 — paralelo (20–25s) FA->>FN: run_funnel_agent FN->>SUP: lee embudo + contacto FN->>OR: identifica etapa FN-->>FA: etapa and FA->>CU: run_contact_update CU->>OR: extrae nombre/datos CU->>SUP: update wp_contactos CU-->>FA: ok end FA->>CA: run_agent (Phase 2) loop hasta 4 iteraciones CA->>OR: invoke + tools OR-->>CA: tool_calls opt scheduling CA->>N8: webhook crear/reagendar/cancelar N8-->>CA: resultado end opt CRM CA->>SUP: update contactos/notas end end CA-->>FA: respuesta + tools_used FA->>SUP: insert wp_mensajes FA->>DBG: stage=run_agent_done FA->>SND: send_message SND->>KP: WhatsAppClient KP->>META: respuesta META->>U: mensaje del bot
Timeouts: funnel 25s · contact_update 20s · LLM por iteración 30s · hard cap 4 iteraciones.
Si la fase 1 falla, la fase 2 corre igual con un prompt mínimo — no se pierde el mensaje.

7. Background loop — retry de mensajes "stuck" background

Cada 5 minutos escanea debug_events buscando mensajes que entraron pero nunca se completaron, y los reintenta hasta 3 veces. Toggle: RETRY_STUCK_ENABLED=false lo desactiva.

10. Despliegue — dual proceso en un contenedor deploy

Un solo Docker image (Node 22 + Python 3.11 venv) corre dos procesos: Express en :3001 (bridge) y Uvicorn en :8000 (FastAPI). El bridge llama al FastAPI por loopback.

Reglas operativas:
  • Trabajar siempre en test → merge a main para producción.
  • No mergear con estrategia ort sin verificar debug_dashboard.py y server.mjs (merges previos borraron imports).
  • docs/ se copia al image → cambios en docs requieren rebuild.
  • Si el bridge crashea, todos los endpoints de WhatsApp caen.
