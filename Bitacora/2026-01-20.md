### Problema Reportado

  - Contactar Administrador (mailto)
  - Reintentar

- Detecta errores de acceso denegado usando `parseAccessDeniedError()`

El registro de nuevos usuarios mediante link de invitación fallaba debido a race conditions:
- El registro en `wp_team_humano` se creaba al ACEPTAR la invitación
- El usuario podía autenticarse ANTES de aceptar

```
DESPUÉS (V2):
1. Crear invitación → wp_team_invitations + wp_team_humano (is_active=FALSE)

3. Aceptar invitación → UPDATE wp_team_humano (is_active=TRUE)
```

## Fix: Error Client-Side al Editar Team Humano

### Problema Reportado
Al editar el email de un miembro del equipo desde otra empresa (siendo rol 1), el sistema mostraba "Application error: a client-side exception has occurred".

### Causa Raíz
Dos bugs en el flujo de actualización:

1. **`TeamMemberModal.tsx`**: El modal cerraba inmediatamente después de llamar `updateMember()` sin verificar si la operación fue exitosa.
2. **`teamStore.ts`**: Si la actualización fallaba por RLS o no existía el registro, `data` era `null` pero se usaba como si fuera válido.

### Correcciones Aplicadas

#### 1. TeamMemberModal.tsx
```typescript
// ANTES: Cerraba sin verificar
await updateMember(memberToEdit.id, formData);
onClose(); // Siempre cerraba

// DESPUÉS: Verifica resultado
const result = await updateMember(memberToEdit.id, formData);
if (result) {
  onClose();
} else {
  // Lee error del store y lo muestra
  const currentError = useTeamStore.getState().error;
  setError(currentError || 'Error al guardar');
}
```

#### 2. teamStore.ts - updateMember()
```typescript
// AÑADIDO: Verificación de data null
if (!data) {
  const errorMsg = 'No se pudo actualizar. Puede que no tengas permisos o el registro no existe.';
  set({ error: errorMsg, isLoading: false });
  return null;
}
```

### Archivos Modificados
- `components/admin/team/TeamMemberModal.tsx`
- `store/teamStore.ts`

### Comportamiento Esperado
- Si falla la actualización, el modal permanece abierto mostrando el error
- Si hay bloqueo por RLS, muestra mensaje descriptivo
- Usuario puede corregir y reintentar sin perder datos del formulario

## Fix: TypeError "Failed to fetch" en Creación de Contactos

### Problema Reportado
Al intentar crear un contacto, aparecía el error genérico: `TypeError: Failed to fetch`

### Causa Raíz
El error ocurre cuando hay problemas de red/conexión con Supabase:
- Sesión de autenticación expirada
- Conexión de red inestable
- Token de autenticación inválido

El cliente de Supabase usa `fetch` internamente, y cuando falla la conexión, lanza este error sin retry automático.

### Solución Implementada

#### 1. Helper Reutilizable `lib/supabase-helpers.ts`
Nuevo archivo con utilidades para operaciones resilientes:
- **`isNetworkError(err)`**: Detecta errores de tipo red/conexión
- **`getNetworkErrorMessage(err)`**: Convierte errores a mensajes amigables
- **`tryRefreshSession()`**: Intenta refrescar la sesión de Supabase
- **`withNetworkRetry(operation, options)`**: Ejecuta operaciones con retry automático
- **`safeSupabaseOperation(operation, options)`**: Wrapper que retorna resultado estructurado

#### 2. Manejo de Errores en `createContact`
- Retry automático (hasta 2 intentos) en errores de red
- Refresh de sesión antes de cada retry
- Backoff exponencial entre reintentos (500ms, 1s)
- Mensaje de error claro: "Error de conexión. Verifica tu internet e intenta de nuevo."

#### 3. Manejo de Errores en `addContactNote`
- Mismo patrón de retry aplicado
- Protección contra errores de red transitorios

### Archivos Creados
- `lib/supabase-helpers.ts` - Helpers reutilizables para errores de red

### Archivos Modificados
- `store/contactStore.ts`:
  - `createContact`: Retry automático con refresh de sesión
  - `addContactNote`: Retry automático con refresh de sesión

### Flujo de Retry
```
Operación → Error de red?
  → Sí → Refresh sesión → Espera 500ms → Retry 1
    → Error de red? → Refresh → Espera 1s → Retry 2
      → Error de red? → Mensaje amigable al usuario
  → No → Retornar error normal de Supabase
```

### Impacto
| Escenario | Antes | Después |
|-----------|-------|---------|
| Red inestable | Error inmediato | 3 intentos automáticos |
| Sesión expirada | Error críptico | Refresh + retry |
| Mensaje de error | "Failed to fetch" | "Verifica tu internet..." |

### Aplicabilidad
El helper `supabase-helpers.ts` puede usarse en otras operaciones críticas:
```typescript
import { withNetworkRetry } from '@/lib/supabase-helpers';

const result = await withNetworkRetry(
  async () => {
    const { data, error } = await supabase.from('tabla').insert(payload);
    if (error) throw error;
    return data;
  },
  { context: 'nombreOperacion', maxRetries: 2 }
);
```
