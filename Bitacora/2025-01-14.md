## Fix: Deep Research "No autorizado"

### Problema
La herramienta Deep Research en el Lab mostraba error "No autorizado" al intentar iniciar una investigación.

### Causa Raíz
Las peticiones fetch en `deepResearchStore.ts` no incluían `credentials: 'include'`, por lo que las cookies de sesión de Supabase no se enviaban al servidor. La API `/api/deep-research` usa autenticación híbrida (cookies + Bearer token) y sin las credenciales, no podía identificar al usuario.

### Solución
Añadido `credentials: 'include'` a ambas llamadas fetch:

```typescript
// POST - Iniciar investigación
const response = await fetch('/api/deep-research', {
  method: 'POST',
  credentials: 'include',  // ← AÑADIDO
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({...})
});

// GET - Polling de estado
const response = await fetch(`/api/deep-research?jobId=${jobId}`, {
  credentials: 'include'  // ← AÑADIDO
});
```

- `store/deepResearchStore.ts` (líneas 130-132, 245-247)

## Fix 2: Deep Research - Middleware bloqueaba antes del Route Handler

### Problema
Después del primer fix, el error "No autorizado" persistía.

### Causa Raíz
El **middleware** (`middleware.ts`) tenía `/api/deep-research` en `protectedApiPaths`, lo que causaba que verificara autenticación ANTES de que la request llegara al route handler.

Si `supabase.auth.getUser()` fallaba en el middleware (por cualquier razón de cookies), retornaba 401 directamente sin ejecutar la lógica robusta del route handler.

### Solución
Mover `/api/deep-research` de `protectedApiPaths` a `publicApiPaths` para que use su propia autenticación interna (igual que `/api/chat`).

### Archivos Modificados
- `middleware.ts` (líneas 62-79)

## Fix 3: Deep Research - Faltaba Authorization Bearer Token

### Problema
Después del segundo fix, seguía el error "Sesión inválida o expirada".

### Causa Raíz
El fetch en `deepResearchStore.ts` no enviaba el **Authorization Bearer token** como lo hace `/api/chat` en `useChatReliable.ts`. Solo tenía `credentials: 'include'` para cookies, pero la API también acepta Bearer token como fallback.

### Solución
Añadir el header `Authorization: Bearer ${accessToken}` a ambos fetch (POST y GET polling):

```typescript
const accessToken = useAuthStore.getState().session?.access_token;

const response = await fetch('/api/deep-research', {
  method: 'POST',
  credentials: 'include',
  headers: { 
    'Content-Type': 'application/json',
    ...(accessToken && { 'Authorization': `Bearer ${accessToken}` })
  },
  body: JSON.stringify({...})
});
```

- `store/deepResearchStore.ts` (imports + líneas 130-140, 252-256)

## Fix: Email Marketing Analytics - Conteo Real de Enviados

### Problema
El dashboard de Analíticas de Email Marketing mostraba máximo **1000** en "Enviados" aunque hubiera más registros.

### Causa Raíz
Supabase tiene un **límite por defecto de 1000 registros** cuando no se especifica un límite explícito. La query original traía todos los registros y los contaba en memoria:

```typescript
// ❌ ANTES - Limitado a 1000 por defecto de Supabase
const { data: sends } = await supabase
  .from('wp_email_envio')
  .select('estado, campana_id, contacto:contacto_id!inner(empresa_id)')
  .eq('contacto.empresa_id', selectedEnterpriseId);

const total = sends?.length || 0;  // Máximo 1000
```

### Solución
Usar queries con `count: 'exact', head: true` para obtener conteos reales directamente de la base de datos:

```typescript
// ✅ DESPUÉS - Conteo real sin límite
const { count: totalCount } = await supabase
  .from('wp_email_envio')
  .select('id, contacto:contacto_id!inner(empresa_id)', { count: 'exact', head: true })
  .eq('contacto.empresa_id', selectedEnterpriseId);
```

### Beneficios
- **Sin límite artificial**: Conteos reales de 0 a millones
- **Mejor rendimiento**: `head: true` no trae datos, solo cuenta
- **Queries paralelas**: Stats por campaña se calculan en paralelo

- `components/admin/email-marketing/AnalyticsTab.tsx` - Función `loadStats()` reescrita
